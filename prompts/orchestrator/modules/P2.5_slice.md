# Phase 2.5S: Vertical Slice Mode（縦切りモード）

## 目的
「まず縦に切ってから機能を拡充」を実現するための軽量発散フェーズ。
フル P2.5（Diverge）の代替として、Slice 対象タスクに適用する。

## 適用条件
以下の **いずれか** に該当する場合、通常の P2.5 ではなくこのモードを使う:
- 新システムの初回実装（end-to-end のスケルトンを通す目的）
- プロトタイプ・検証（本番品質は不要）
- インターフェースが未確定で変更が見込まれる

## 手順（通常 P2.5 より軽量）

### Step 1: ゴール明文化（1文）
- 「この Slice で何が end-to-end で動くようになるか」を1文で書く。
- 例: 「1P カメラで猫が移動し、3P に切り替えても追従する」

### Step 2: 最小パス特定
1. ゴールを達成する最小のコード変更パスを特定する。
2. 「触るファイル/クラス」を列挙する（5個以下に抑える）。
3. 5個を超える場合、Slice を分割する。

### Step 3: リスク1行（Devil's Advocate 簡易版）
- 「これが動かない最大の理由は？」を1つだけ挙げ、対策を1行で書く。

### Step 4: テスト方針
- Test Phase = **Slice** を明記。
- ビルド成功 + 手動検証が必須。詳細テストは後続 Stable フェーズで追加。

## スキップされる項目（通常 P2.5 との差分）
- 3案比較（1案で十分なら不要）
- Impact Radar（最小パスで影響範囲は自明）
- Bird's Eye View Mermaid（省略可）

## 完了条件
- Slice ゴールが1文で明文化されている
- 最小パスが5ファイル以下で特定されている
- リスク1件と対策が記録されている
- Test Phase = Slice が明記されている

## Slice 終了判定（いつ通常モードに戻るか）
以下のいずれかを満たしたら、後続タスクは通常の P2.5（Diverge）に戻る:
- public API / インターフェースが確定し、2タスク以上変更なく使えた
- end-to-end の最小パスが動作している
- Orchestrator が「Slice 完了、Stable へ移行」を宣言した

## 次フェーズ
P3（分割と戦略）— ただし Slice モードでは Worker 数は原則1、並列化しない。
